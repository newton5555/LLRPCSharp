# ADR 0001：结构化日志与原始帧观测

- 状态：Accepted
- 日期：2026-07-23

## 背景

SDK 同时提供面向业务的高级 Reader API 和面向协议的底层 API。一次高级操作可能产生多条 LLRP 请求、响应和异步事件，因此调用方需要在使用高级封装时，仍能观察真实的底层通信，并将这些通信与发起它们的高级操作关联起来。

诊断需求分为两类：

1. 面向人员和日志系统的可读、可筛选结构化日志；
2. 面向抓包、审计和回放的无损原始帧数据。

两类需求在性能、数据保真度、生命周期和敏感信息处理方面不同，不应由同一个接口承担。

## 决定

### 日志抽象

底层组件统一依赖 `Microsoft.Extensions.Logging` 抽象。组合根接收 `ILoggerFactory`，各组件使用独立类别创建日志；组件内部也可以按 .NET 约定接收 `ILogger<T>`。

未配置日志时使用 `NullLoggerFactory` 或 `NullLogger<T>`，调用方无需提供日志实现，SDK 也不直接依赖具体日志提供程序。

日志采用结构化属性，不通过拼接字符串承载可查询字段。常用字段包括：

- `ReaderId` 或 Endpoint；
- `OperationId` 与 Operation；
- Direction（`TX`/`RX`）；
- MessageType 与 MessageId；
- FrameLength；
- ElapsedMs；
- Status、ExceptionType 和错误码。

### 高级操作关联

每次高级操作在入口创建唯一的 `OperationId`，并通过 `ILogger.BeginScope` 建立包含 `OperationId`、Operation 和 Reader 标识的日志作用域。该上下文沿异步调用链传递，使 Session、Transaction、Codec 和 Transport 产生的日志自动带有相同的关联信息。

一个高级操作触发多条请求时，这些请求共享同一个 `OperationId`；每条协议消息仍以自己的 MessageId 区分。由读写循环独立接收、且无法归属于某个高级调用的异步 Reader 事件，不强行继承错误的操作作用域。

### 真实报文边界

原始 wire bytes 在 Transport 边界观测：

- 发送方向记录实际交给网络写入逻辑的完整 LLRP 帧；
- 接收方向记录从 TCP 字节流组装出的完整 LLRP 帧，并保留解析前的原始字节；
- 不允许将解码对象重新编码后冒充接收到的原始报文；
- TCP 分片日志属于可选的更底层诊断，不替代完整 LLRP 帧观测。

### 可读日志与无损观测分离

`ILogger` 仅用于可读诊断日志。完整 Hex dump 可以作为显式启用的 Trace 日志，但日志管道不提供无损、按序、可靠交付保证，也不作为抓包或回放数据源。

无损抓包、持久化和回放使用独立的 `ILlrpFrameObserver` 抽象。观察事件至少携带方向、时间、连接标识和真实帧字节，并明确字节缓冲区的所有权与有效期。观察器不得在 Transport 的关键读写路径中无限阻塞；具体的缓冲、背压、丢弃或持久化策略由实现和配置明确约定。

Transport 对观察器采用以下失败与并发契约：

- 传入的帧内存在观察回调完成前有效；需要异步持有时，观察器必须复制或在回调内可靠接管数据；
- 同一连接上的发送观察顺序与实际发送顺序一致，接收与发送方向的回调可能并发；观察事件携带递增的 Connection Generation，用于区分同一 Transport 重连前后的不同 TCP 字节流；
- 网络操作一旦成功，观察器异常只记录为诊断告警，不反向把已经提交的协议命令报告成“发送失败”，避免上层重试造成重复命令；
- 组合观察器逐个隔离失败并继续投递给后续观察器，全部投递后再报告单个或聚合异常，避免一个抓包接收器故障连带屏蔽日志/指标接收器；
- 需要无损落盘的实现应在回调返回前把帧可靠接纳到自身队列，并独立暴露队列/持久化健康状态。`ILlrpFrameObserver` 是原始帧边界，不把任意日志提供程序升级为可靠存储。

### 日志级别

- `Information`：连接状态、高级操作开始/完成及最终结果；
- `Debug`：消息摘要、MessageId、协议状态、事务匹配和耗时；
- `Trace`：完整 TX/RX Hex、解码结构以及按需启用的 TCP 分片；
- `Warning`：超时、可恢复的非法输入、未知消息或参数、重连等异常状态；
- `Error`：导致操作或连接失败的传输、编码和解码错误。

高频路径在格式化或复制报文前检查对应日志级别，避免日志关闭时产生不必要的分配。

### 敏感字段策略

完整 Hex 和完整解码结构默认关闭。日志默认只输出消息摘要和非敏感元数据。

Access Password、Kill Password、认证材料以及扩展协议中的敏感载荷必须在可读日志中脱敏；不得仅依赖日志提供程序进行事后过滤。如果原始帧无法在保持协议真实性的同时安全脱敏，则只有调用方显式选择并知晓风险后，才允许输出完整 Hex。

`ILlrpFrameObserver` 提供的是原始数据，可能包含敏感信息。启用观察器属于显式的安全决策，其存储权限、保留期限、传输加密和访问审计由宿主负责，并应在公开配置与文档中提示。

## 后果

### 正面后果

- 使用高级 Reader API 时仍可追踪实际的底层请求和响应；
- 日志能接入现有 `Microsoft.Extensions.Logging` 生态，无需 SDK 自建日志框架；
- `OperationId`、MessageId 和日志 Scope 能同时表达业务操作与协议事务关系；
- 原始帧在正确边界采集，避免重新编码导致的失真；
- 可读诊断与抓包/回放职责分离，双方可以独立演进。

### 代价与约束

- 所有异步层必须正确传播调用上下文，后台接收循环需要避免误用操作 Scope；
- Trace Hex 和帧观察会带来额外复制、吞吐和存储成本，必须按需启用；
- 引入 `ILlrpFrameObserver` 后需要明确缓冲区所有权、背压和异常隔离语义；
- 原始帧可能包含机密信息，宿主必须为抓包数据承担额外的安全与合规责任。

## 不采用的方案

### 自定义通用日志接口

不采用。它会重复 `Microsoft.Extensions.Logging` 已有的级别、类别、Scope、结构化属性和提供程序集成能力。

### 仅在高级 API 记录日志

不采用。高级 API 无法准确呈现最终发送和实际接收的字节，也无法覆盖协议级调用和后台事件。

### 仅使用 `ILogger` 实现抓包和回放

不采用。日志提供程序可能过滤、采样、截断、重排或异步丢弃事件，不能保证二进制数据的完整性与顺序。
